Highlight #1 Centralized Version Control Systems

A centralized version control system is a server that has all the versions of a webpage, that is central to everyone who is working on it. This way, everyone knows what's going on, and administrators have a say on who is able to edit what. This biggest downside is that everyone is connected to one server, so if that server goes down, all of their versions go down as well. Also, if you keep all of the information on that one server, if it goes down, you lose everything, not just parts of it.

Highlight #2 The 3 Stages of git Commands

The three main stages are commited, data is saved on your local database, modified, the file is modified but not commited to the database, and staged, which means that you've made sure that you want that file to be commited. The three main section are the repository, staging area and working directory. The repository stores all the screenshots and databases for your project, the staging area holds the information about which files you are going to commit, and the working directory is the area where you modify and use the files for the project.

Highlight #3 Distributed Version Control Systemsr

Instead of looking at the last snapshots of the files, the people working on the project have different copies of the whole project on their local server. That way if a server is down, then they can just copy it from one of the other copies. This is a better system than a centralized version control system because then you have a much less chance of losing all of your work.
